## 0. Purpose & Scope

This specification defines the internal integration framework for EGNOS and Galileo PNT services within UTS domain 010 — Mission Systems.

For certification and operational approval purposes, this document distinguishes explicitly between:

1. **Certifiable navigation baseline** — SBAS / EGNOS Safety-of-Life functions intended to support RNAV GNSS operations to LPV or related minima under the applicable recognised external regulatory framework.
2. **Supplemental non-creditable functions** — Galileo Open Service (OS), OSNMA and High Accuracy Service (HAS) capabilities used for resilience, performance enhancement, cyber-resilience, engineering analysis, or research / validation unless and until explicit operational credit is approved by the competent authority.
3. **Internal governance constructs** — CCTLS tokens, CLB bindings, UTS lifecycle artefacts and programme-level entities such as ESSA, which do not replace external certification authorities, service declarations, or recognised standards.

This specification covers:

1. **PNT source characterisation** — engineering parameters used to define, monitor and validate EGNOS and Galileo service usage within UTS domain 010.
2. **SBAS airborne compliance and approval path** — internal evidence path for SBAS-capable equipment, installation, validation and operational approval interfaces.
3. **Dual-constellation interoperability** — GPS + Galileo architecture, signal fusion, RAIM/FDE support and signal selection logic.
4. **EGNOS SoL service usage** — requirements applicable where EGNOS SoL is the credited integrity source for approach or related operations.
5. **Galileo supplemental service usage** — OS, OSNMA and HAS requirements where these services are used as supplemental capabilities.
6. **CCTLS token types and evidence** — tokenised internal governance artefacts generated by this specification.

**Governance note:** internal institutional references such as ESSA, EACST, UTS and CCTLS are programme-governance constructs unless explicitly identified as formally recognised external authorities. They shall not be used to imply external regulatory competence where such competence belongs to recognised authorities, approved standards and declared service providers.

1. **airborne equipment performance standards**,

## 4. Dual-Constellation Interoperability (GPS + Galileo)

### 4.1 Architecture

The dual-constellation navigation solution combines GPS and Galileo within a tightly or loosely coupled architecture as defined by the installed system.

The combined solution may improve availability, geometry and robustness. Unless specifically approved, these improvements shall be treated as supplemental and shall not be used to replace the certified integrity function credited to EGNOS SoL.

```text
┌─────────────────────────────────────────────────────┐
│           Navigation Processing Unit (NPU)          │
│                                                     │
│  ┌──────────────┐        ┌──────────────┐           │
│  │ GPS Receiver │        │ Galileo Rx   │           │
│  └──────┬───────┘        └──────┬───────┘           │
│         │                        │                   │
│         └──────────┬─────────────┘                   │
│                    │                                 │
│           ┌────────▼────────┐                        │
│           │ Fusion / Select │                        │
│           │ / Integrity     │                        │
│           └────────┬────────┘                        │
│                    │                                 │
│      ┌─────────────▼─────────────┐                   │
│      │ EGNOS SoL interface       │                   │
│      │ (credited only where      │                   │
│      │ explicitly approved)      │                   │
│      └─────────────┬─────────────┘                   │
│                    │                                 │
│      ┌─────────────▼─────────────┐                   │
│      │ Supplemental interfaces   │                   │
│      │ OSNMA / HAS / IMU / INS   │                   │
│      └─────────────┬─────────────┘                   │
│                    │                                 │
│             Position / Velocity / Time               │
└─────────────────────────────────────────────────────┘
```

### 4.2 Interoperability Requirements

| Requirement            | Parameter                                | Policy                                                           |
| ---------------------- | ---------------------------------------- | ---------------------------------------------------------------- |
| Common time reference  | GPS-Galileo time offset handling         | Must be applied, bounded and verified by installed-system design |
| Common geodetic datum  | WGS-84 / GTRF alignment                  | Datum handling must be explicit in the solution                  |
| SIS compatibility      | Signal compatibility and frequency plan  | Must be validated at receiver / system level                     |
| Dual-frequency use     | Iono-free combinations where implemented | Engineering function; not automatically creditable               |
| Fault detection        | Cross-constellation monitoring           | Supplemental unless tied to approved architecture                |
| Signal selection logic | Geometry / weighting / masking           | Must be deterministic and documented                             |
| Continuity support     | Combined constellation availability      | Treated as supplemental unless explicitly approved               |

### 4.3 RAIM / FDE Requirements

RAIM / FDE shall be implemented where required by the applicable equipment architecture and operational concept.

For LPV / APV safety claims, RAIM / FDE shall be treated as complementary to, not a substitute for, the certified SBAS integrity function.

Minimum internal validation expectations:

* RAIM availability assessed for intended operational area,
* FDE behaviour assessed under representative fault scenarios,
* alert latency measured and documented,
* false alert / missed detection assumptions recorded in validation evidence,
* geometry degradation cases included in simulation and flight-test planning.

### 4.4 Output Segregation

The installed architecture shall segregate:

* **credited integrity output**, derived only from the approved SoL path where applicable;
* **supplemental positioning enhancements**, derived from Galileo OS, HAS, multi-constellation fusion or other non-creditable paths;
* **cyber-resilience outputs**, such as OSNMA status or spoofing indicators.

## 5. EGNOS Safety-of-Life (SoL) Service Requirements

### 5.1 SoL Service Role

The EGNOS SoL service is the only service within this specification intended to support **creditable** navigation integrity claims for LPV / APV-type operations, subject to:

* approved airborne equipment,
* approved aircraft installation,
* approved procedures and operations,
* applicable service declaration and service status.

It provides:

* SBAS augmentation and integrity information within its declared scope,
* protection-level information for approved use cases,
* service-status information subject to applicable notices, declarations and operational constraints.

### 5.2 SoL Operational Requirements

| Requirement                                       | Value / Policy                                                                                                                             | Claim Class              |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------ |
| SoL receiver / installation basis                 | Must comply with the applicable approved equipment and aircraft integration basis                                                          | CREDITABLE               |
| Internal AMPEL360 target architecture             | Tier Beta-3 minimum                                                                                                                        | INTERNAL_ONLY            |
| GEO / SBAS tracking                               | Must satisfy the approved receiver / installation basis                                                                                    | CREDITABLE               |
| Service area / applicability confirmation         | Must be reviewed using current service-status information                                                                                  | CREDITABLE               |
| Misleading information / integrity interpretation | Must follow the applicable declared and approved basis                                                                                     | CREDITABLE               |
| Continuity interpretation                         | Must follow the time basis, geographical scope and service commitment defined by the applicable SoL service declaration and approval basis | CREDITABLE               |
| LPV / APV protection-level monitoring             | Must remain within approved thresholds for the relevant operation segment                                                                  | CREDITABLE               |
| Reversionary navigation                           | Must be available where required by approval basis                                                                                         | CONDITIONALLY_CREDITABLE |

### 5.3 SoL Service Monitoring

Programmes using EGNOS SoL shall implement:

1. **Pre-flight service-status review** — all applicable EGNOS service notices, operational notices and related service-status information within the defined planning window shall be reviewed and documented.
2. **In-flight SoL monitoring** — the receiver / navigation system shall monitor SoL integrity messages continuously and execute the defined reversion or missed-approach logic when required.
3. **Service-status recording** — the pre-flight and dispatch decision path shall be archived in `GNS-010-EV-006`.
4. **Reversionary navigation** — a non-GNSS or alternate approved navigation path shall be defined and demonstrated wherever required by the approval basis.

### 5.4 Credit Rule

No OSNMA, Galileo OS, Galileo HAS or generic dual-constellation availability improvement may be used to substitute the credited EGNOS SoL integrity function unless a separate explicit approval basis exists.

## 6. Galileo Supplemental Services

## 6.1 Galileo Open Service (OS)

Galileo OS may be used as a freely accessible positioning source for engineering, mission, supplemental navigation or resilience purposes, subject to the installed architecture and operational concept.

It shall be treated in this specification as a **supplemental** service unless a separate explicit approval basis exists.

### 6.1.1 OS Requirements

| Requirement                                       | Policy                                                        | Claim Class                               |
| ------------------------------------------------- | ------------------------------------------------------------- | ----------------------------------------- |
| Receiver support                                  | Must support the selected OS signals used by the architecture | INTERNAL_ONLY                             |
| Performance measurement                           | Must be measured and documented in intended operational area  | INTERNAL_ONLY                             |
| Use in en-route / terminal support                | Permitted as supplemental input                               | NON_CREDITABLE unless separately approved |
| Use in SoL integrity                              | Not permitted as substitute for EGNOS SoL integrity           | NON_CREDITABLE                            |
| Integration with spoofing / resilience monitoring | Recommended where mission assurance requires it               | INTERNAL_ONLY                             |

## 6.2 Galileo OSNMA

OSNMA may be used as a cyber-resilience enhancement for Galileo Open Service message authentication.

It shall not be interpreted as:

* a substitute for certified navigation integrity,
* an operational approval in itself,
* a replacement for approved SoL monitoring.

### 6.2.1 OSNMA Requirements

| Requirement                           | Policy                                                                          |
| ------------------------------------- | ------------------------------------------------------------------------------- |
| OSNMA receiver support                | Recommended where anti-spoof resilience objectives apply                        |
| Authentication latency                | Record and validate in cyber test evidence                                      |
| Key / cryptographic material handling | Must follow the applicable distribution and configuration process               |
| Logging                               | Authentication status shall be available to the cyber / mission assurance layer |
| Operational credit                    | No direct navigation-integrity credit unless separately approved                |

## 6.3 Galileo High Accuracy Service (HAS)

Galileo HAS is an optional high-accuracy augmentation service that may support engineering, supplemental navigation-performance enhancement, research or validation use cases.

Unless specifically approved, HAS shall not be used as the credited integrity source for safety-of-life guidance.

### 6.3.1 HAS Engineering Targets

| Parameter                      | Target / Measurement Policy                        | Claim Class   |
| ------------------------------ | -------------------------------------------------- | ------------- |
| Horizontal accuracy            | Record measured performance in validation evidence | INTERNAL_ONLY |
| Vertical accuracy              | Record measured performance in validation evidence | INTERNAL_ONLY |
| Convergence time               | Record and validate in `GNS-010-EV-007`            | INTERNAL_ONLY |
| Signal availability            | Record as service / measurement evidence           | INTERNAL_ONLY |
| Frequency / decoder capability | Must match installed receiver architecture         | INTERNAL_ONLY |

### 6.3.2 HAS Integration Requirements

| Requirement            | Detail                                                                                                                                                                                           | Claim Class                                      |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------ |
| Receiver E6 capability | Required where HAS is implemented                                                                                                                                                                | INTERNAL_ONLY                                    |
| Convergence management | Precision-enhancement phase shall not be entered until convergence criteria are satisfied                                                                                                        | INTERNAL_ONLY                                    |
| HAS + EGNOS SoL hybrid | Permitted only as a segregated supplemental architecture: HAS may enhance accuracy estimates, while any credited SoL guidance function remains solely dependent on the approved integrity source | NON_CREDITABLE for HAS contribution to integrity |
| HAS failure management | Automatic fallback to approved or predefined alternate navigation mode required                                                                                                                  | INTERNAL_ONLY                                    |
| Latency                | Validate message and update-rate behaviour in evidence                                                                                                                                           | INTERNAL_ONLY                                    |

### 6.3.3 Restriction

Galileo HAS may only be used as a **supplemental high-accuracy input** after confirmed convergence, unless a separate approval basis explicitly authorises primary operational credit.

### 12.2 Governance Note

ESSA is referenced in this document as a proposed programme-level institutional concept. Unless formally established and externally recognised, ESSA shall not be interpreted as a competent authority, certification body, or substitute for applicable recognised regulatory institutions and standards.

### 12.3 Definitions

| Term                          | Definition                                                                                                                                                       |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Creditable**                | A function or claim that may support certification or operational approval, subject to an applicable recognised basis and completed evidence                     |
| **Non-creditable**            | A function or claim that may be used for engineering, resilience or supplemental performance enhancement, but not as direct support for SoL certification claims |
| **Internal-only**             | A programme governance or engineering construct with no direct external regulatory standing                                                                      |
| **Service-status review**     | Pre-flight or planning-window review of applicable service notices, notices to operators and declared availability information                                   |
| **Supplemental architecture** | An architecture in which a non-creditable source may enhance performance estimates without replacing the credited integrity source                               |
